## Sprint rule

> **1 Sprint = 1 clear product objective.**

**Sprint 71.1 objective:** Introduce **Runtime Resolver** as the single decision point for engine selection. Migrate Pipeline, Doctor, and execution toward the Runtime Kernel **without breaking** legacy APIs.

**Prerequisite:** Platform Sprint **71** (10-capability catalog) + Sprint **70.9** (engine execution history).

**Note:** Platform Sprint **71** (`TASK-0071`) delivered the expanded capability catalog (33 engines). Sprint **71.1** delivers **convergence** — the missing architectural piece identified in the Runtime/Pipeline audit.

---

## Problem statement

After Sprints 70.x, Lumen has a **complete Runtime platform** (catalog, discovery, readiness, provisioning, benchmarks, analytics, dashboard) but the **video pipeline still uses a parallel legacy stack**:

| Legacy (Pipeline) | Runtime (Platform) |
| ----------------- | ------------------ |
| `AIEngineRegistry` | `EngineCatalogDefinitions` |
| `GET /api/ai/providers` | `GET /api/runtime/*` |
| `pipeline_configuration` (PostgreSQL) | `configuration.json` |
| Provider IDs (`faster_whisper`) | Catalogue IDs (`faster_whisper_large_v3`) |
| Env vars as “current” | Selection store + resolve policy |

Principle:

```text
Runtime decides.
Pipeline orchestrates.
Worker executes.
UI observes.
```

# Slices

## P71.1-SLICE-01 — Runtime Resolver

**Domain types**

| Type | Responsibility |
| ---- | -------------- |
| `RuntimeResolveRequest` | Capability + context |
| `RuntimeResolveContext` | Language, duration, planner preference, consumer |
| `ResolvedEngine` | Decision: engineId, reason, confidence, blocked, adapterKey |
| `EngineExecutionPlan` | planId, adapterKey, parameters, fallbackPlan |
| `RuntimeResolveReason` | user_selection, planner_context, hardware_recommended, … |
| `RuntimeFallbackPlan` | Alternate engine when primary blocked |

**Services**

- `RuntimeResolverInterface` — port
- `RuntimeResolver` — implementation (`Infrastructure/Runtime/Kernel/`)
- `EngineAdapterRegistry` — catalogue ID ↔ legacy provider key

**Resolution policy (priority)**

1. Manual selection (`configuration.json`, mode `manual`)
2. Planner context (`preferredEngineId`)
3. Hardware recommendation (`HardwareReportBuilder`)
4. Profile recommendation (`RecommendationEngine`)
5. Ops bootstrap (env-configured engine)
6. Catalog default
7. Fallback alternative when blocked (explicit, logged)

**API:** `POST /api/runtime/resolve`

**Tests:** unit tests for policy ordering and adapter mapping.

---

## P71.1-SLICE-02 — Capability Contract

**Deliverables**

- `PipelineStageCapabilityMapper` — bijective map `PipelineStageType` ↔ `EngineCatalogCapability` (6 video stages)
- `RuntimeCapability` / `EngineCatalogCapability` remain the public capability namespace
- Future capabilities (OCR, Vision, …) have no pipeline stage — exposed only via Runtime

**Rule:** Pipeline code references **capabilities** or **stages**; never hardcoded engine IDs.

---

## P71.1-SLICE-03 — Runtime Selection Store

**Unify (transition)**

| Source | Target role |
| ------ | ----------- |
| `configuration.json` | **SSOT** for profile, selectionMode, manualSelections |
| `pipeline_configuration` | Legacy API compatibility (dual-write) |
| Env vars | Ops bootstrap only (not user “current”) |
| Planner overrides | Context enrichment per job, not persisted catalogue |

**Deliverables**

- `RuntimeSelectionSynchronizer` — on `PUT /api/pipeline`, sync manual selections → Runtime config
- Migration mapping: `faster_whisper` → `faster_whisper_large_v3`, etc.

**Not in this sprint:** remove PostgreSQL `pipeline_configuration` table.

---

## P71.1-SLICE-04 — Pipeline Settings via Runtime

**Frontend**

- `PipelineBuilder` loads `GET /api/runtime/capabilities/{capability}/selection-view` per stage
- `PipelineStageSelector` displays: Recommended, Current, Installed, Blocked reason
- Dropdown options derived from installed engines + adapter mapping

**Removed dependency:** `GET /api/ai/providers` from Pipeline Settings (legacy API remains for AI Engine Settings page until deprecation sprint).

---

## P71.1-SLICE-05 — Execution Adapter Layer

**Transform (strangler)**

```text
AIProviderResolver (façade)
  → if RUNTIME_KERNEL_UNIFIED: EngineExecutionAdapter → RuntimeResolver
  → else: legacy pipeline_configuration + AIEngineRegistry
```

**Deliverables**

- `EngineExecutionAdapter::planForStage()` / `legacyProviderIdForStage()`
- Provider factories **unchanged** — only their input source changes
- Blocked engine → explicit exception (no silent fallback)

**Feature flag:** `RUNTIME_KERNEL_UNIFIED` (default `true` in `services.yaml`).

---

## P71.1-SLICE-06 — Doctor via Runtime

**`scripts/doctor.sh`**

- Before: hardcoded `command -v` for 5 binaries + Ollama
- After: `GET /api/runtime/readiness` + per-capability `selection-view` for video pipeline
- Retains: `/health`, `/ready`, `/live`, `/api/platform/readiness`

---

## P71.1-SLICE-07 — Planner as Runtime Client

**`DeterministicPipelinePlanner`**

- Injects `RuntimeResolverInterface`
- Enriches resolve context with video intelligence preferences
- Does **not** read `AIEngineRegistry` for final decision when kernel unified

**Contract:** Planner suggests → Runtime resolves.

## P71.1-SLICE-10 — UI Consistency

**Criterion:** Pipeline Settings and Runtime Dashboard capability cards use the same semantic fields:

- referenceEngineId / referenceDisplayName
- recommendedEngineId / recommendedDisplayName
- currentEngineId / currentDisplayName
- installedEngineIds
- blockedReason / blocked / executable

**Source:** `RuntimeResolver::capabilitySelectionView()` and `RuntimeDashboardAssembler::buildCapabilities()` must stay aligned (shared resolver logic for “current”).
