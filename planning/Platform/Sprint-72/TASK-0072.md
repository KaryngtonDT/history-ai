## Problem statement

After Sprint 71.1, Lumen has a unified **Runtime Resolver** and Pipeline/Doctor convergence, but engine management remains fragmented:

| Gap (pre-72) | Sprint 72 deliverable |
| ------------ | --------------------- |
| Provisioning APIs without lifecycle façade | `RuntimeEngineLifecycleService` + managers |
| No central engine manager UI | `/settings/runtime/engines` — Provision Center |
| Global selection mode only | **Auto / Manual / Locked** per capability |
| Resolver returns engine ID only | `ResolvedEngineIntelligence` — reason, confidence, alternatives, estimates |
| Doctor hardcoded binary checks | `GET /api/runtime/doctor` — SSOT |
| Shadow cannot explain engine choices | `runtimeContext` in Watch context |
| No lifecycle notifications | `RuntimeNotificationService` |

Principle:

```text
Runtime decides.
Worker executes.
UI observes.
```

# Slices (11 parts)

## P72-SLICE-01 — AI Engine Lifecycle Management (Part 1)

**Services** (`Infrastructure/Runtime/Lifecycle/`)

| Class | Responsibility |
| ----- | -------------- |
| `RuntimeEngineLifecycleService` | Façade: install, update, repair, remove, validate, benchmark, enable, disable |
| `RuntimeProvisionManager` | Install — venv, models, dependencies, readiness |
| `RuntimeUpdateManager` | Engine version updates |
| `RuntimeRepairManager` | Repair misconfigured engines |
| `RuntimeRemovalManager` | Remove / enable / disable |
| `RuntimeVersionManager` | Installed version metadata |
| `RuntimeDependencyManager` | Dependency graph per engine |
| `RuntimeModelManager` | Model/checkpoint metadata |

**API**

| Method | Path | Purpose |
| ------ | ---- | ------- |
| POST | `/api/runtime/engines/{id}/install` | Install engine |
| POST | `/api/runtime/engines/{id}/update` | Update engine |
| POST | `/api/runtime/engines/{id}/repair` | Repair engine |
| DELETE | `/api/runtime/engines/{id}` | Remove engine |
| POST | `/api/runtime/engines/{id}/validate` | Revalidate readiness |
| POST | `/api/runtime/engines/{id}/test` | Benchmark single engine |

Install flow: provision → readiness → benchmark → analytics registration → notification.

## P72-SLICE-02 — Runtime Provision Center (Part 2)

**Route:** `/settings/runtime/engines` — **AI Engine Manager**

**Frontend**

- `RuntimeEnginesPage` — page shell
- `RuntimeProvisionCenter` — capability sections + engine cards

**API:** `GET /api/runtime/engines/management` — assembled by `RuntimeEngineManagementAssembler`

**Per capability (10):** Speech-to-Text, Translation, TTS, Voice Clone, Lip Sync, Video Render, Vision, OCR, Embeddings, Reranking (+ catalog extensions).

**Engine card fields:** name, capability, provider, version, reference/recommended/current badges, installed/ready/blocked/mock status, hardware compatibility, benchmark score, avg duration, avg accuracy, blocked reason, documentation, install command.

**Actions:** Install, Validate, Benchmark, Repair, Remove.

---

## P72-SLICE-03 — Runtime Engine Selection (Part 3)

**Domain:** `CapabilitySelectionMode` — `auto`, `manual`, `locked`

**Configuration** (`RuntimeConfiguration`)

| Field | Role |
| ----- | ---- |
| `capabilityModes` | Per-capability mode override |
| `manualSelections` | User pick when Manual |
| `lockedSelections` | Fixed engine when Locked |

**Resolver policy** (`RuntimeResolver::pickEngineId`)

1. **Locked** → `lockedSelections[capability]` (never auto-switch)
2. **Manual** → `manualSelections[capability]`
3. **Auto** → planner → hardware → profile → ops → catalog default

**API:** `PUT /api/runtime/selection` with `capabilityModes`, `manualSelections`, `lockedSelections`

**UI:** Auto / Manual / Locked toggle per capability in Provision Center; radio pick for Manual/Locked.

---

## P72-SLICE-04 — Runtime Resolver Intelligence (Part 4)

**Service:** `RuntimeResolverIntelligence`

**Inputs:** hardware profile, capability, media duration, language, execution history, user profile, resolve reason.

**Output:** `ResolvedEngineIntelligence` attached to `ResolvedEngine`

| Field | Description |
| ----- | ----------- |
| `alternativeEngineId` | Second-best option |
| `estimatedDurationSeconds` | From analytics or context |
| `expectedAccuracy` | Historical success rate |
| `explanation` | Human-readable decision rationale |
| `confidence` | On parent `ResolvedEngine` |

**API:** `POST /api/runtime/resolve` returns full plan + intelligence.

---

## P72-SLICE-05 — Runtime Learning (Part 5)

**Foundation:** Sprint 70.9 `EngineStatisticsAggregator` + `DurationPredictionEngine`

**Recorded per execution:** duration, prediction error, success/failure, engine version, hardware profile, language, media duration.

**Integration:** `RuntimeResolverIntelligence::findAnalytics()` feeds resolver; management assembler surfaces stats on engine cards.

**Rule:** Every pipeline execution improves duration estimates and fastest-engine rankings.

---

## P72-SLICE-06 — Runtime Recommendations (Part 6)

**Services**

| Class | Role |
| ----- | ---- |
| `RecommendationEngine` | Profile-based per-capability picks |
| `RuntimeRecommendationProfilesService` | Multi-profile comparison |

**API:** `GET /api/runtime/recommendations/profiles`

**Profiles:** Best Quality, Fastest, Lowest RAM, Hardware Recommended, Current Selection.

**Used by:** management assembler, doctor report, Shadow `runtimeContext.recommendationProfiles`.

---

## P72-SLICE-07 — Runtime Analytics (Part 7)

**Route:** `/settings/runtime/analytics` (Sprint 70.9 dashboard, integrated in management narrative)

**API:** `/api/runtime/analytics/*` via `RuntimeAnalyticsController`

**Surfaces:** engine rankings, execution history, prediction accuracy, capability statistics, duration/success/failure rates.

**Link:** Provision Center secondary nav → Analytics.

---

## P72-SLICE-08 — Runtime Doctor (Part 8)

**API:** `GET /api/runtime/doctor` — `RuntimeDoctorReportService`

**Script:** `scripts/doctor.sh` consumes doctor API (no hardcoded engine list)

**Report sections:** installed, missing, blocked, mock, capabilities (executable/current/recommended), readiness counts, recommendations.

**Principle:** Doctor **only** consumes Runtime — no parallel binary checks.

---

## P72-SLICE-09 — Shadow Integration (Part 9)

**Builder:** `RuntimeShadowContextBuilder`

**Injected into:** `GetShadowContextHandler` → `WatchContextResult.runtimeContext`

**Payload:** selection, recommendationProfiles, doctorSummary, per-capability current/recommended engines, `promptHints` for common engine questions.

**Shadow can answer:** which engine to use, why Runtime chose it, install feasibility, blocked reasons, performance comparisons.

---

## P72-SLICE-10 — Notifications (Part 10)

**Service:** `RuntimeNotificationService` — JSON store, max 100 items

**Events:** `engine_installed`, `engine_updated`, `engine_validation_failed`, benchmark completion

**API:** `GET /api/runtime/notifications`

**Recorded by:** `RuntimeEngineLifecycleService` on install/update outcomes.
