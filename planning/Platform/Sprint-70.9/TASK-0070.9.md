# TASK-0070.9 — Engine Performance Analytics & Adaptive Duration Estimation

## Phase

**Platform — Runtime Learning & Pipeline UX**

## Status

**Planned** (after Sprint 70.8 background pipeline orchestration)

## Constitutional reference

[docs/vision/LUMEN_VISION_2030.md](../../../docs/vision/LUMEN_VISION_2030.md)

## Problem statement

Today Lumen shows **static** duration estimates (rules + hardware heuristics). Users see start time and remaining time, but:

- no **estimated completion date**
- no **actual completion date** or **actual duration** after the fact
- no **estimation accuracy** feedback loop
- no **per-engine benchmark history** tied to real pipeline jobs
- Shadow cannot explain *why* today's estimate differs from last week
- Runtime cannot recommend the fastest engine **for this hardware** based on **your** measurements

Principle:

> **Every completed job becomes a benchmark. Estimates improve automatically.**

## Relationship to existing systems

| Existing | Scope | Sprint 70.9 extension |
| -------- | ----- | --------------------- |
| `ExecutionHistory` | Per-video pipeline version snapshots (quality, config, optimization) | **Complement, do not replace** — keep for replay/audit |
| `BenchmarkRunner` | Host engine synthetic benchmarks (`make runtime-benchmark`) | Link optional `benchmarkScore`; production truth = job history |
| `PipelineStageDurationEstimator` | Rule-based estimates at job creation | Becomes **fallback** when history is sparse |
| `PipelineJob` | `startedAt`, `estimatedDurationSeconds`, `elapsedSeconds` | Source events for history recording |
| Runtime Center (`/settings/runtime`) | Health, capabilities, provisioning | Add **analytics** tab at `/settings/runtime/analytics` |

## Vision

Transform Lumen into a platform that **continuously learns** engine performance from every pipeline execution. The UI exposes prediction **and** reality so users can compare engines and trust the runtime.

## P70.9-SLICE-02 — Adaptive Estimation Engine

**Services:** `DurationPredictionEngine`, `PredictionHistory`, `EngineStatistics`

Per `(stage, engineId, hardwareProfile[, model])` compute:

- average, median, P90 duration
- fastest / slowest execution
- average estimation error
- prediction confidence (function of sample count + variance)
- **relative speed score** (normalized vs fastest engine on same profile for that stage)

**Algorithm**

1. If ≥ `N_min` samples (configurable, default 3): predict from rolling window (median or P75 conservative).
2. Else if media duration known: blend historical **speed factor** (sec/media sec) with `PipelineStageDurationEstimator` fallback.
3. Else: pure rule fallback (current behavior).

**Deliverables**

- Replace direct `PipelineStageDurationEstimator` usage in orchestrator with `DurationPredictionEngine::estimateForStage()`
- `EngineStatisticsAggregator` — refresh stats after each recorded execution
- Persist aggregated stats in `engine_statistics` table (or materialized view)
- PHPUnit: prediction improves after synthetic history seed

## P70.9-SLICE-06 — Runtime Learning Loop

After every completed/failed stage:

1. `RecordEngineExecutionHandler`
2. `EngineStatisticsAggregator::refresh`
3. `DurationPredictionEngine::invalidateCache` (if cached)
4. Update runtime dashboard capability scores / engine ranking where applicable
5. Feed recommendation engine (reuse Runtime 70.4 selection hooks)

**Deliverables**

- Messenger async handler optional for aggregation (sync OK for v1)
- Integration test: complete STT job → history row → next estimate changes
